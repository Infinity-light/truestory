## 模块设计

### 模块 1: 会议房间（创建与加入）

**职责**：让主持人创建会议、生成入会凭证（六位会议号 + 深链接）；让另两人通过凭证进入；三人各自连接钱包并签"开始"消息后会议正式开始。

**不做**：超过三人的会议、密码保护、会议预约。

**上游依赖**：Supabase（meetings 表 + Auth）、RainbowKit（钱包连接 UI）、Monad testnet（钱包链确认）。

#### 后端

**数据模型**（Supabase Postgres）：

```sql
-- meetings 表
create table meetings (
  id uuid primary key default gen_random_uuid(),
  room_code text unique not null check (char_length(room_code) = 6),  -- 六位数字
  host_address text not null,                                          -- 主持人钱包地址 (0x...)
  status text not null default 'waiting',                              -- waiting / starting / recording / reviewing / signing / sealed
  created_at timestamptz not null default now(),
  expires_at timestamptz not null default now() + interval '30 minutes',
  recording_started_at timestamptz,
  recording_ended_at timestamptz,
  on_chain_tx_hash text                                                -- 上链后填入
);
create index idx_meetings_room_code on meetings(room_code);
create index idx_meetings_status on meetings(status);

-- participants 表
create table participants (
  meeting_id uuid not null references meetings(id) on delete cascade,
  wallet_address text not null,
  role text not null check (role in ('host', 'participant')),
  joined_at timestamptz not null default now(),
  start_sig text,                                                       -- 开始签名（钱包对会议号+nonce 的签名）
  start_signed_at timestamptz,
  end_sig text,                                                         -- 结束共识签名
  end_signed_at timestamptz,
  review_completed bool not null default false,
  primary key (meeting_id, wallet_address)
);
create index idx_participants_meeting on participants(meeting_id);
```

**API 路径**（Next.js App Router 全部走 `/api/...`）：

| 方法 | 路径 | 请求体 | 响应 | 错误码 |
|------|------|--------|------|--------|
| POST | /api/meetings | `{hostAddress}` | `{meetingId, roomCode, joinUrl}` | 400 缺地址、409 钱包未连接 |
| POST | /api/meetings/[roomCode]/join | `{walletAddress}` | `{meetingId, currentParticipants[]}` | 404 不存在、409 已满、410 过期 |
| POST | /api/meetings/[roomCode]/sign-start | `{walletAddress, signature, signedMessage}` | `{startedAt}` 或 `{waitingFor: [addr]}` | 400 签名无效、404 不在会议中 |
| GET | /api/meetings/[roomCode] | - | `{meeting, participants[]}` | 404 |

**业务逻辑**（伪代码）：

```
POST /api/meetings:
  validate hostAddress is 0x... format
  generate roomCode = 6 random digits, retry if conflict
  insert into meetings (room_code, host_address, status='waiting')
  insert into participants (meeting_id, wallet=hostAddress, role='host')
  return roomCode + joinUrl

POST /api/meetings/[code]/join:
  fetch meeting by room_code
  if not exist -> 404
  if status != 'waiting' -> 409
  if participants count >= 3 -> 409
  insert participant (role='participant')
  emit Supabase Realtime broadcast: 'participant_joined'
  return current state

POST /api/meetings/[code]/sign-start:
  validate signature using viem.verifyMessage
  signedMessage must be: `TriSign Start: roomCode=${code} ts=${ts}`
  update participants.start_sig
  if all 3 participants have start_sig:
    update meetings.status = 'recording'
    update meetings.recording_started_at = now()
    emit Supabase Realtime: 'meeting_started'
  else:
    emit Supabase Realtime: 'participant_signed_start'
```

### 模块 2: 录音与实时 STT 文字瀑布流

**职责**：三人各自在浏览器录音，音频流式送阿里云 STT，转写文字通过 Supabase Realtime 广播给会议中所有三人，按时间序排列成文字瀑布流。

**不做**：音视频传输、说话人识别（每人手机各录各的天然分离）、消息持久化压缩。

**上游依赖**：模块 1（meeting 状态）、阿里云百炼 STT API、Supabase Realtime。

#### 后端

**数据模型**：

```sql
create table messages (
  id uuid primary key default gen_random_uuid(),
  meeting_id uuid not null references meetings(id) on delete cascade,
  speaker_address text not null,
  original_text text not null,                  -- STT 原始输出
  final_text text,                              -- 审阅修改后；null 表示未修改，等同 original_text
  spoken_at timestamptz not null,               -- 客户端时间戳（毫秒精度）
  server_received_at timestamptz not null default now(),
  original_hash text not null,                  -- keccak256(speaker + original_text + spoken_at)
  final_hash text,                              -- keccak256(speaker + final_text + spoken_at)；null 时 = original_hash
  is_disputed bool not null default false       -- 任一其他人标过异议
);
create index idx_messages_meeting_time on messages(meeting_id, spoken_at);

create table disputes (
  message_id uuid not null references messages(id) on delete cascade,
  disputer_address text not null,
  disputed_at timestamptz not null default now(),
  primary key (message_id, disputer_address)
);
```

**API 路径**：

| 方法 | 路径 | 用途 |
|------|------|------|
| POST | /api/stt/token | 后端为前端签发阿里云 STT 临时 token（避免暴露 API key） |
| POST | /api/messages | 持久化一条 STT 转写消息（前端 STT 出字后立即调用） |
| WS | wss://[stt-proxy] | 后端 STT 代理 WebSocket，转发音频流到阿里云，转发转写回前端 |

**业务逻辑**：

```
POST /api/stt/token:
  validate session has joined a meeting in 'recording' status
  generate阿里云 STT 临时 token (服务端用 AccessKey 调阿里云签发接口)
  return {token, wsEndpoint, expiresIn}

POST /api/messages:
  validate session.walletAddress matches body.speakerAddress (防伪)
  insert into messages
  compute originalHash = keccak256(speaker || original_text || spoken_at)
  emit Supabase Realtime broadcast on channel `meeting:${roomCode}:messages`
```

**阿里云 STT 接入方式**：
- 后端启动一个 WebSocket 代理服务（Next.js API Routes 不支持 long-lived WS，需用 Vercel Edge Functions + Cloudflare Durable Objects 或者**单独部署一个 Node WS 服务**到 Render/Railway）
- 妥协方案（推荐）：前端直接调阿里云 STT WebSocket，token 由 /api/stt/token 签发（避免暴露 AccessKey）。无需自建 WS 代理

### 模块 3: 审阅与修改（含异议标注）

**职责**：让三人在会议结束后独立审阅完整记录，每人能修改自己说的话（修正 STT 错误），对别人的话标异议；三人都完成审阅后自动进入签名阶段。

**不做**：修改他人内容、对修改后内容再标异议、版本回滚。

**上游依赖**：模块 2（messages 数据）。

#### 后端

**API 路径**：

| 方法 | 路径 | 用途 |
|------|------|------|
| PATCH | /api/messages/[id] | 修改消息内容（仅 speaker 本人） |
| POST | /api/messages/[id]/dispute | 标异议（非 speaker 本人） |
| DELETE | /api/messages/[id]/dispute | 撤销异议（仅在 reviewing 状态且 speaker 未修改时） |
| POST | /api/meetings/[code]/complete-review | 标记本人审阅完成 |

**业务逻辑**：

```
PATCH /api/messages/[id]:
  validate session.walletAddress == message.speaker_address
  validate meeting.status == 'reviewing'
  validate participant.review_completed == false (一旦完成审阅不能再改)
  update messages set final_text=body.text, final_hash=keccak256(...)
  emit Realtime broadcast: 'message_updated'

POST /api/messages/[id]/dispute:
  validate session.walletAddress != message.speaker_address
  validate meeting.status == 'reviewing'
  validate participant.review_completed == false
  insert into disputes
  update messages set is_disputed=true
  emit Realtime: 'message_disputed'

POST /api/meetings/[code]/complete-review:
  update participants set review_completed=true where wallet=session
  if all 3 review_completed:
    update meetings.status = 'signing'
    emit Realtime: 'all_reviewed'
```

### 模块 4: 共识签名与上链

**职责**：三人各自钱包签"我对最终版负责"消息，签名后前端将完整数据打包写入 Monad 测试网合约，链上保留每笔签名 + 终 hash + 异议 mapping。

**不做**：每条消息独立上链（只整体上链一次）、原版 hash 上链（按用户决策中间版方案，原 hash 只存 Supabase）。

**上游依赖**：模块 3（审阅完成状态 + 最终消息列表）、Monad testnet RPC + 钱包。

| POST | /api/meetings/[code]/submit-chain | 三人都签完后，由后端代理上链调用合约（gas 由后端钱包支付）或返回前端自己上链 |

**合约设计**（`contracts/TriSign.sol`）：

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

contract TriSign {
    struct Signature {
        address signer;
        uint256 signedAt;
        bytes32 finalMessagesRoot;   // 最终消息列表 hash root
        bytes32 disputesRoot;         // 异议 mapping hash
        bytes sig;                    // 钱包签名原始字节
    }

    struct Meeting {
        bytes32 roomCodeHash;
        address[3] participants;
        Signature[3] startSigs;       // 开始签名
        Signature[3] endSigs;          // 结束共识签名
        bytes32 finalMessagesRoot;
        bytes32 disputesRoot;
        uint256 sealedAt;
        bool isSealed;
    }

    mapping(bytes32 => Meeting) public meetings;  // key = meetingId hash

    event MeetingStarted(bytes32 indexed meetingId, address[3] participants, uint256 timestamp);
    event SignatureSubmitted(
        bytes32 indexed meetingId,
        address indexed signer,
        bytes32 finalMessagesRoot,
        bytes32 disputesRoot,
        bytes sig
    );
    event MeetingSealed(bytes32 indexed meetingId, uint256 timestamp);

    function startMeeting(bytes32 meetingId, address[3] calldata participants, Signature[3] calldata startSigs) external {
        // 验证 startSigs 是 participants 对 meetingId 的有效签名
        // 存入 mapping
        // emit MeetingStarted
    }

    function submitConsensus(
        bytes32 meetingId,
        bytes32 finalMessagesRoot,
        bytes32 disputesRoot,
        bytes[3] calldata sigs
    ) external {
        // 三人的 sigs 一次性提交
        // 验证每个 sig 是对应 participant 对 (finalMessagesRoot || disputesRoot) 的签名
        // 设置 isSealed = true
        // emit SignatureSubmitted x 3 + MeetingSealed
    }

    function verifyMeeting(bytes32 meetingId, bytes32 candidateMessagesRoot)
        external view returns (bool isValid, address[3] memory signers)
    {
        Meeting storage m = meetings[meetingId];
        return (m.finalMessagesRoot == candidateMessagesRoot && m.isSealed, m.participants);
    }
}
```

**业务逻辑**：

```
finalMessagesRoot = keccak256(abi.encodePacked(
  message_1_hash, message_2_hash, ..., message_n_hash
))
其中 message_i_hash = keccak256(speaker || final_text || spoken_at_ms)

disputesRoot = keccak256(abi.encodePacked(
  (message_id_1, disputer_1), (message_id_2, disputer_2), ...
))

POST /api/meetings/[code]/sign-end:
  validate signature using viem.verifyMessage
  signedMessage = `TriSign End: roomCode=${code} finalRoot=${finalRoot} disputesRoot=${disputesRoot}`
  update participants.end_sig
  if all 3 have end_sig:
    update meetings.status = 'sealed'
    return {ready: true, allSigs: [...]}
  else:
    return {ready: false, waitingFor: [...]}

POST /api/meetings/[code]/submit-chain (前端调用方案 A：前端自己调合约):
  返回 {meetingId, participants, finalRoot, disputesRoot, sigs[]} 让前端打包成合约调用
  前端用 wagmi useWriteContract 调 TriSign.submitConsensus(...)

或方案 B（后端代理上链）:
  后端用环境变量里的 deployer 钱包私钥（仅 demo 用）调合约，前端不需要再签 tx
  // 但这样三人都不真的"花 gas 上链"，故事感弱
  // 推荐方案 A
```

**链选择**：方案 A —— 由发起 submitConsensus 的人（通常是主持人）支付 gas，其他两人只签消息不上链。这样合约只调用一次，gas 成本可接受。

### 模块 5: 事后验证

**职责**：让任何人拿到会议记录 JSON + 交易哈希，能在 /verify 页验证记录没被改过。

**不做**：异议验证（demo 范围内只验内容 hash 一致性）、错误消息精确定位每个字（按消息级别定位即可）。

├── VerifyButton
└── ResultDisplay
    ├── SuccessBlock (绿色对勾 + 元数据)
    └── FailureBlock (红色叉 + 不匹配消息列表)

**业务逻辑**：

```
1. 用户上传 JSON：{messages: [...], roomCode, participants, finalRoot}
2. 用户输入 tx hash
3. 点"验证"按钮：
   a. 前端从 JSON 重新计算 finalMessagesRoot = keccak256(message_hashes)
   b. 用 wagmi readContract 调 TriSign.verifyMeeting(meetingId, candidateRoot)
   c. 比对结果，渲染 Success 或 Failure
   d. Failure 时遍历每条消息，重算单条 hash，标红不匹配的消息
```
