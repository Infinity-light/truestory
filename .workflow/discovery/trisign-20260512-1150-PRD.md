# PRD: TriSign — 三方共识会议记录 App（黑客松版）

## 1. 目标

让三个分散在不同地点的人在浏览器里开一次"有共识效力"的会议——会议结束后产生一份三方签字、不可篡改、连分歧都诚实记录的会议纪要，任何人未来都能链上验证它没被改过。

黑客松场景下，team 自身就是 demo 主角：三人在台上各开自己的电脑/手机，演示完整流程——开会、录音、文字瀑布、互改、签名、上链，让评委直接看到"如果将来有人争议谁先提出了这个百亿想法，链上记录能证伪"。

## 2. 约束与边界

**必须满足**：
- 网页 App 形态（HTTPS 部署，无原生 App），跨平台浏览器（Mac Safari、iOS Safari、Chrome、Android Chrome 均支持）
- 一天黑客松内可演示
- 全程除钱包外不需要安装任何东西
- 会议内容上链可验证（包括异议、原版与终版双 hash）

**明确不做**：
- App 内不传输音视频——三人靠 STT 文字瀑布流"看到"对方讲话，听不到对方声音
- 不做生物特征签名（WebAuthn / Face ID 全不做），只用钱包签名
- 不做"删除消息"功能——一旦上链所有消息都保留
- 不允许任何人修改他人消息（只能标异议）
- 不做超过三人的会议（demo 锁定三人场景）
- 不实现"被篡改演示"的现场交互——demo 只演正向路径，篡改流程由口头讲

**演示降级预案**：如果 demo 前 4 小时实时 STT 仍跑不通，临场切到"录音结束后才统一转录"版本，剧本台词预先写好，让评委感知不到降级。

## 3. 用户操作动线

主持人（角色 H）：
1. 进入 App 首页 → 连接钱包 → 点击"新建会议"
2. 拿到一个会议号（六位数字）+ 一个深链接 URL
3. 通过任意渠道（微信、Telegram、口头）把会议号或链接分享给另两人
4. 等其他两人加入 → 看到三人都"已连接钱包"后点击"开始会议" → 钱包弹窗签名"我准备好开会了"
5. 等另两人也签完"开始" → 进入录音状态
6. 对着自己的麦克风讲话 → 屏幕上的文字瀑布流不断冒出三人各自的字
7. 会议讲完后点击"结束会议"
8. 进入审阅页 → 看到完整时间序文字记录 → 修改自己说错的话 / 对别人的话标异议 → 点击"完成审阅"
9. 等另两人也完成审阅 → 点击"确认共识签名" → 钱包弹窗签名"这就是我说的，我对此负责"
10. 三人都签完 → 看到"已上链"提示和交易哈希链接

参与者（角色 P）：
1. 收到会议号或链接 → 点开链接（自动进入会议）或在 App 首页输入会议号
2. 连接钱包 → 看到"主持人 H 已就位"和自己已就位的状态
3. 接下来的流程与主持人 5-10 步完全相同

验证者（角色 V，可以是任何第三方）：
1. 拿到会议记录文件（JSON）+ 链上交易哈希
2. 打开 App 的"验证"页 → 上传文件 / 粘贴交易哈希
3. 立即看到结果：绿色"✓ 一致" 或 红色"✗ 已篡改" + 指出篡改位置（如果一致性失败）

## 4. 模块设计

### 模块 1: 会议房间（创建与加入）

**职责**（用户视角）：让三个人通过网页找到同一个会议空间，并各自用钱包证明"我是我"。

**核心功能**：
- 主持人创建会议、生成会议号和链接
- 其他人通过会议号或链接进入
- 三人各自连接钱包
- "开始会议"双重门槛：三人都连钱包 + 三人都签"开始"消息

**页面/交互设计**：
- **首页**：两个大按钮——"新建会议"和"加入会议"。右上角"连接钱包"按钮（未连接时灰色禁用其他操作）
- **新建会议页**：六位会议号大字显示在中央，下方一个可复制的链接，再下方实时显示"已就位：[H 钱包地址简写]"列表，开始按钮在三人到齐前灰色禁用
- **加入会议页**：输入会议号的输入框 + 加入按钮；或者直接通过链接进入跳过此页
- **会议候机页**（三人都在但还没全签"开始"时）：实时显示三人头像（钱包地址简写）+ 状态徽章（未连接 / 已连接 / 已签开始），所有人都"已签开始"时自动进入下一个页面

**信息架构**：
- L0 概览：首页（新建 / 加入二选一）
- L1 聚焦：候机页（看谁来了、谁没来）
- L2 管理：会议号/链接分享区
- L3 执行：钱包签名弹窗

**关键业务规则**：
- 会议号有效期 30 分钟（避免被路过的人乱用）
- 已签"开始"的人退出，会议状态回到"待开始"
- 第四个人尝试加入时看到"会议已满（三人）"提示

**验收标准**：
- **成功路径**（三人各开浏览器顺利进入）
  - Given 主持人在 /create 页连了钱包并点击"新建会议"
  - When 拿到的会议号被另两人在各自浏览器的 /join 页输入并点击"加入"，三人都连了钱包并点击"开始会议"，三人都在钱包弹窗中点确认
  - Then 在所有三人的浏览器中，2 秒内自动跳转到 /meeting/[会议号]/recording 录音页面，页面顶部显示"会议进行中 · 三人在线"
- **失败路径**（第四人加入）
  - Given 三人已经签完"开始"进入了录音状态
  - When 第四个浏览器尝试在 /join 页输入同一个会议号并点击"加入"
  - Then 页面显示红色提示"该会议已满（三人会议不支持更多参与者）"，没有跳转
- **边界路径**（一人不签"开始"，会议永远不开始）
  - Given 三人都已连接钱包但只有两人签了"开始"
  - When 等待超过 30 秒
  - Then 候机页一直停留，未签的人头像区显示橙色"等待 [钱包地址] 签名…"，开始会议按钮持续禁用

### 模块 2: 录音与实时 STT 文字瀑布流

**职责**（用户视角）：让三人各自对着自己的麦克风讲话，所有人的话变成实时滚动的文字消息流，三人都能看到。

**核心功能**：
- 浏览器获取麦克风权限并录音
- 各人手机/电脑各自录制并送 STT
- STT 转写实时回到所有三人的浏览器，按时间戳排序
- 文字瀑布流以微信聊天式样式呈现，每条消息显示发言人钱包地址简写 + 时间戳 + 内容

**页面/交互设计**：
- 顶部状态栏：会议号、三人在线状态徽章、"录音中"红点闪烁、时长计时
- 主体区域：文字瀑布流，每条消息一个气泡，自己的话气泡靠右（绿色）、别人的话靠左（白色），每条带时间戳和钱包地址简写
- 底部固定按钮："结束会议"（只有主持人可点）
- 麦克风权限第一次被请求时由浏览器原生弹窗处理，不需 App 内部包装

**信息架构**：
- L0 概览：消息总流 + 状态栏
- L1 聚焦：每条消息（点击查看时间戳精度毫秒级、句子是哪段录音对应的）
- L3 执行：底部"结束会议"按钮

**关键业务规则**：
- 麦克风权限被用户拒绝时显示"必须开启麦克风才能继续，请点这里"按钮唤起浏览器权限再请求
- STT 引擎选型由 Planning 决定，但要求支持中英文混合识别（demo 上你和 Javi 会讲中英混合）
- 消息只追加，不允许在录音过程中修改或删除——录音中只看不改
- 三人讲话时间戳冲突（同时讲）的消息按各自手机本地时间戳处理，前端按时间戳全局排序

**验收标准**：
- **成功路径**（讲话出字）
  - Given 三人都在 /meeting/[ID]/recording 页面，每人浏览器已被授予麦克风权限
  - When 用户 A 对自己设备说出一句话（如"我觉得这个想法不错"）
  - Then 5 秒内所有三人的浏览器瀑布流底部都出现一条新消息，消息包含 A 的钱包地址简写、ISO 时间戳、文字"我觉得这个想法不错"，消息气泡在 A 自己的视图中靠右绿色，在 B 和 C 视图中靠左白色
- **失败路径**（麦克风权限被拒）
  - Given 用户 B 进入录音页时浏览器弹出麦克风权限请求并被点击"拒绝"
  - When 页面加载完成
  - Then 页面中央显示红色提示"麦克风权限被拒，无法录音"，下方有一个"重新授权"按钮，点击后浏览器再次弹出权限请求
- **边界路径**（录音中尝试修改消息）
  - Given 录音正在进行中，瀑布流已有 5 条消息
  - When 用户长按或右键点击自己的某条消息
  - Then 不出现任何修改菜单（录音中只读），最多出现"复制"功能

### 模块 3: 审阅与修改（含异议标注）

**职责**（用户视角）：让三人在会议结束后独立审阅完整记录，每人能修正自己说错的话或被 STT 转写错的话，能对别人的话表达不同意见。

**核心功能**：
- 三人各自看到同一份完整记录
- 每人能修改自己的任何一条消息（修正 STT 错误或追加补充）
- 每人能对别人的任何一条消息标"异议"
- 不能对修改后的消息再标异议——发言人改了之后他人无异议权
- 完成审阅按钮：点完后此人就不能再改/再标，等待其他人

**页面/交互设计**：
- 顶部进度条：三人审阅状态（未开始 / 审阅中 / 已完成）
- 主体：完整消息列表，按时间序排列
- 自己的消息：右上角有"编辑"小图标，点击进入行内编辑模式，原版用浅灰删除线显示在修改版上方
- 别人的消息：右上角有"标异议"小图标，点击切换"已标异议"状态，已标异议的消息左侧出现橙色竖条
- 底部："完成审阅"按钮（不可逆，点完弹确认对话框"确认后无法再修改本人发言"）

**信息架构**：
- L0 概览：完整消息列表 + 三人审阅状态条
- L1 聚焦：单条消息（看修改前后对比）
- L2 管理：编辑模式 / 异议切换
- L3 执行：完成审阅按钮（不可逆操作）

**关键业务规则**：
- 自己的消息修改后，"原版"作为对比预览在自己视图里可见；其他人**只看到最终版**，不知道发言人修改过什么（按用户决策"不能针对修改打异议"）
- 异议是消息级布尔属性，不带具体异议理由（demo 范围内）
- 三人都"完成审阅"后自动进入下一阶段（共识签名）
- 完成审阅按钮点击后再次进入此页面变成只读状态

**验收标准**：
- **成功路径**（自己改自己的话）
  - Given 用户 A 在 /meeting/[ID]/review 页面，瀑布流中有自己的一条消息"我觉的这个想法不错"（STT 转错了"觉得"）
  - When A 点击自己消息右上角的编辑图标，把"觉的"改成"觉得"，按回车保存
  - Then 该消息显示更新后的文字"我觉得这个想法不错"，在 A 的视图中上方有浅灰删除线版本"我觉的这个想法不错"，在 B 和 C 的视图中只显示"我觉得这个想法不错"，不显示原版
- **失败路径**（试图改别人的话）
  - Given 用户 B 在审阅页看到 A 的消息
  - When B 双击 A 的消息或试图通过任何 UI 入口编辑 A 的消息
  - Then 不出现任何编辑入口，最多出现"标异议"按钮，B 点击"标异议"后 A 的消息左侧出现橙色竖条
- **边界路径**（B 标了异议之后 A 修改自己的话）
  - Given B 已对 A 的某条消息标了异议（消息左侧已有橙色竖条）
  - When A 编辑该消息内容并保存
  - Then 消息内容更新，**橙色异议竖条保留**（业务上"B 的异议是针对消息事件本身的，不因内容修改而消除"），但 B 不能再修改异议状态（按"不能针对修改打异议"规则）

### 模块 4: 共识签名与上链

**职责**（用户视角）：三人独立审阅完成后，各自用钱包签一次"我对最终版负责"的消息，签名上链留下永久证据。

**核心功能**：
- 三人各自看到最终版会议记录的简要摘要 + 关键统计（消息数、异议数、自己改过几条）
- 各自钱包签名"我对此版本共识"
- 等三人都签完后，前端将完整数据（消息+异议+签名）打包写入合约
- 显示上链成功后的交易哈希链接

**页面/交互设计**：
- 顶部：会议摘要框，显示"共 N 条消息 · M 条带异议 · 你修改了 K 条"
- 中部：三人头像和签名状态（未签 / 签名中 / 已签 ✓）
- 下方："确认共识签名"按钮，点击触发钱包弹窗
- 签名完成后：等其他人页面，三人都签完后自动进入"上链中..."状态
- 上链成功页：大绿色对勾 + 交易哈希（可复制 + 链上浏览器跳转链接）

**关键业务规则**：
- 签名内容包含：会议号、最终消息列表的 hash、原始消息列表的 hash、异议列表的 hash、自己的钱包地址
- 链上记录最终保留：每个人的签名（含上述完整内容）、每条消息的原版 hash 和终版 hash、所有异议
- 链选择由 Planning 决定，要求 testnet（建议 Base Sepolia / Polygon Mumbai / Solana Devnet 三选一基于团队熟悉度）
- 任一人签名失败（钱包拒绝）则等待重试，不会因为一人卡住就回到审阅页

**验收标准**：
- **成功路径**（三人都签且上链）
  - Given 三人在 /meeting/[ID]/sign 页面，三人都未签
  - When 三人依次点击"确认共识签名"按钮，并在钱包弹窗中各自点击"签名"
  - Then 三人的浏览器都跳转到 /meeting/[ID]/done 页面，页面顶部显示大绿色对勾 + "已上链"，下方显示交易哈希（如 "0xab12...3f89"），点击交易哈希跳转到链浏览器对应交易页
- **失败路径**（钱包签名拒绝）
  - Given 用户 A 在签名页点击"确认共识签名"，钱包弹出
  - When A 在钱包中点击"拒绝"
  - Then 页面显示红色提示"签名已取消"，按钮变回可点状态供 A 重新发起；B 和 C 的页面继续等待 A，不被影响
- **边界路径**（其中一人钱包断网）
  - Given 用户 B 钱包已签名，但提交时网络中断
  - When B 重新连上网络
  - Then B 的浏览器自动重试上链，无需重新签名（签名已在本地缓存）

### 模块 5: 事后验证

**职责**（用户视角）：让任何第三方拿到会议记录文件后能验证它没被改过。

**核心功能**：
- 上传会议记录 JSON 文件 / 输入交易哈希
- 前端重新计算消息 hash 并与链上对比
- 显示验证结果（一致 / 不一致 + 指出哪部分被改了）

**页面/交互设计**：
- 验证页 /verify：上传文件区 + 交易哈希输入框（二选一）
- 验证结果区：大绿色对勾"记录与链上一致"或大红色叉"记录与链上不一致"
- 不一致时下方显示具体哪条消息的 hash 不匹配，标红展示

**关键业务规则**：
- 验证流程纯前端 + 链上读取，无后端介入
- 任何人无需登录即可访问

**验收标准**：
- **成功路径**（正版文件验证一致）
  - Given 用户 V 拿到一份未篡改的会议记录 JSON 文件 + 交易哈希
  - When V 在 /verify 上传文件并粘贴交易哈希，点"验证"
  - Then 3 秒内页面显示绿色对勾 + "✓ 此记录与链上一致，会议时间 [ISO 时间]"
- **失败路径**（篡改后文件被识破）
  - Given 用户 V 拿到一份被人改过其中一条消息的会议记录 + 原交易哈希
  - When V 上传并点验证
  - Then 页面显示红色叉 + "✗ 此记录与链上不一致"，下方标红列出"消息 #N（钱包 0x12...3f 的消息）hash 不匹配"
- **边界路径**（demo 时间口头讲不演交互）
  - 黑客松 demo 不在现场触发此模块，只在 PPT/口头讲到"我们还做了验证页，篡改后能识破"，模块本身要求开发完成但 demo 不演

## 5. 失败兜底（演示风险预案）

如果 demo 前 4 小时存在以下情况，按顺序降级：

| 风险 | 触发条件 | 降级方案 |
|------|----------|----------|
| 实时 STT 推送跑不通 | demo 前 4 小时 WebSocket 推送仍丢字/延迟超过 10 秒 | 切到"录音结束后才统一转录"模式，主持人结束按钮触发批量 STT，演示时台词预先写好让评委感知不到 |
